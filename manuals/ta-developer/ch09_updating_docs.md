# Chapter 9: Updating Documentation

## Documentation System

The docs are built with MkDocs and the Material theme. Source files are in `docs/*.md`, configured by `mkdocs.yml`.

Live site: https://t-o-m-tool-oauto-mationator.github.io/scpi-instrument-toolkit/

## Editing Documentation

### 1. Edit markdown files

All doc source files are in `docs/`. Each instrument type has its own page:

    docs/psu.md         # Power supply commands
    docs/dmm.md         # Multimeter commands
    docs/scope.md       # Oscilloscope commands
    docs/scripting.md   # Scripting reference
    docs/examples.md    # Bundled examples

### 2. Code fence annotations

Always annotate code blocks with a language identifier:

**Wrong:**

    ```
    psu set 1 5.0
    ```

**Right:**

    ```text
    psu set 1 5.0
    ```

Use `text` for REPL commands/output, `python` for Python code, `bash` for shell commands.

### 3. Variable syntax in docs

- Use `{var}` (not `${var}`)
- Use `var = value` (not `set var value`)
- Use `print "message {var}"` (with quotes)

### 4. Update mkdocs.yml nav

When adding a new page, add it to the `nav` section in `mkdocs.yml`:

    nav:
      - REPL:
        - My New Page: my_new_page.md

## Building Locally

Preview the docs before deploying:

    mkdocs build

This builds the site to `lab_instruments/site/` (which is gitignored). Check for warnings in the output.

To preview live with hot-reload:

    mkdocs serve

Opens a local server at http://127.0.0.1:8000.

## Deploying

Deploy to GitHub Pages:

    mkdocs gh-deploy --force

This builds and pushes to the `gh-pages` branch. The site updates within a few minutes.

Always deploy after committing doc changes -- never leave docs undeployed.

## Validation Scripts

### validate_doc_examples.py

Checks that docs/examples.md stays in sync with the EXAMPLES dict:

    python scripts/validate_doc_examples.py

Reports:
- PASS/FAIL -- whether each example runs successfully in mock mode
- SYNC/DRIFT/MISSING -- whether docs match the code

Run this after changing `examples.py` or `docs/examples.md`.

### generate_doc_screenshots.py

Regenerates the terminal SVG screenshots in `docs/img/terminal/`:

    python scripts/generate_doc_screenshots.py

Run this after changing any REPL output format or adding new screenshot scenarios. The script runs commands in mock mode and renders the colored output as SVG using the `rich` library.

## The @docs-sync Subagent

Use this Claude Code subagent after any change to docs:

    @docs-sync

It runs a 4-phase process:
1. Audit staleness (finds outdated content)
2. Fix safe issues
3. Validate examples with --mock
4. Build and deploy

## Adding a New Documentation Page

1. Create `docs/my_new_page.md`
2. Add it to `mkdocs.yml` nav
3. Add code fence annotations (always use language identifiers)
4. Build locally: `mkdocs build`
5. Fix any warnings
6. Deploy: `mkdocs gh-deploy --force`

## Auto-Generated Reference Pages

Three reference pages are auto-generated from source code:

- **Instruments & Capabilities** -- generated by `scripts/gen_ref_pages.py`
- **REPL Command Reference** -- generated from command handler docstrings
- **Expression Functions** -- generated from safe_eval function list

These update automatically on `mkdocs build`. Do not edit them manually.

## Common Documentation Tasks

### Adding a new instrument page

1. Create `docs/mytype.md` with command reference
2. Add tab content for each supported model (use `=== "Model Name"` syntax)
3. Add to `mkdocs.yml` nav under the REPL section
4. Add to `docs/instruments.md` supported models table
5. Build and deploy

### Adding terminal screenshots

1. Edit `scripts/generate_doc_screenshots.py`
2. Add a new scenario to the SCENARIOS list
3. Run the script to regenerate all screenshots
4. Embed in docs: `![Title](img/terminal/filename.svg)`

### Updating the API reference

API reference pages in `docs/api/` are auto-generated from docstrings. To update them:

1. Improve the docstring in the driver source file
2. Run `mkdocs build` -- the reference page updates automatically
